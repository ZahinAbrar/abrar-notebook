Exploring Agentic AI

Building a paper-reviewing agent from a simple tool-caller to a production multi-agent system ‚Äî one level at a time.
About this series

I wanted a concrete, end-to-end project to understand how agentic AI systems actually get built ‚Äî not just the theory, but the real decisions around tool design, memory, multi-agent orchestration, and productionization. So I built one: a paper-reviewing agent that I gradually extended across five levels of increasing capability. Each level adds exactly one layer of complexity ‚Äî no more ‚Äî so you can see what each piece actually buys you. The project is built in Python (Google Colab), using the Agno framework and OpenAI models. Click a level below to explore.

Level 1 ‚Äî Agent with Tools

What I'm building here

The simplest useful agent isn't a chatbot ‚Äî it's something that actually does things: reads files, writes outputs, calls utilities. Level 1 is about that baseline. I give a language model a set of tools and a task ‚Äî "read this paper, write a structured review" ‚Äî and watch it plan and execute steps without me holding its hand.

The paper I used as a test case is Denoising Diffusion Probabilistic Models (Ho et al., 2020). But the agent is general: swap in any PDF and it handles it the same way.

The full pipeline. No memory, no retrieval ‚Äî just tools, a model, and a task.

There are exactly three components:

  • PDF extractor ‚Äî pulls raw text out of the PDF page by page using pypdf, writes it to paper.txt in the Colab workspace.
  • Agent ‚Äî an agno.Agent with CodingTools attached. CodingTools gives it the ability to read files, write files, and run small Python snippets in the workspace.
  • Workspace ‚Äî a local directory (/content/workspace) that serves as shared state between the extractor and the agent.

The tools

The key design choice at this level is what tools to give the agent. I didn't build custom tools ‚Äî I used CodingTools from Agno, which wraps read/write/exec operations into a standard tool interface the model can call. The agent sees these as: read_file(path), write_file(path, content), run_python(code), and list_files(path).

That's all it needs. The model decides on its own when to read, when to write, and what to put in the review. I don't script the steps ‚Äî I describe them in the system prompt and let the model figure out the ordering.

Step 1 ‚Äî extract text from PDF

pdf ‚Üí paper.txtpython
from pathlib import Path
from pypdf import PdfReader

WORKSPACE = Path("/content/workspace")
WORKSPACE.mkdir(exist_ok=True)

pdf_path = WORKSPACE / "paper.pdf"    # upload your PDF here first
reader   = PdfReader(str(pdf_path))

chunks = []
for i, page in enumerate(reader.pages):
    t = page.extract_text() or ""
    chunks.append(f"\n\n===== PAGE {i+1} =====\n{t}")

paper_txt = "\n".join(chunks)
(WORKSPACE / "paper.txt").write_text(paper_txt, encoding="utf-8")
print("Extracted chars:", len(paper_txt))

I delimit each page with a header (===== PAGE N =====) so the agent ‚Äî and I, when debugging ‚Äî can tell which part of the paper a given paragraph comes from. When the model reads this file, those headers become natural anchors for things like "find the evaluation section."

Step 2 ‚Äî define the agent

agent definitionpython
from agno.agent import Agent
from agno.models.openai import OpenAIResponses
from agno.tools.coding import CodingTools

agent = Agent(
    name="PaperReviewer",
    model=OpenAIResponses(id="gpt-4o-mini"),
    tools=[CodingTools(workspace_dir=str(WORKSPACE))],
    instructions=(
        "You are a paper reviewing agent.\n"
        "You MUST use tools.\n"
        "Step 1: Open and read paper.txt from the workspace.\n"
        "Step 2: Write a structured review to review.md in the workspace.\n"
        "Step 3: Verify by listing files.\n\n"
        "Review format:\n"
        "# Paper Review\n"
        "## Summary (3-4 sentences)\n"
        "## Contributions (bullet list)\n"
        "## Strengths (bullet list)\n"
        "## Weaknesses (bullet list)\n"
        "## Questions for Authors (bullet list)\n"
        "## Score: X/10\n\n"
        "Be critical but fair. Focus on empirical evaluation."
    ),
)

A few choices worth explaining. I used gpt-4o-mini rather than a larger model ‚Äî it's cheap enough to iterate on, and for a well-structured task like "read this file and write in this format" it performs just as well. The instructions are tightly scoped: I enumerate the steps explicitly and specify the exact markdown sections I want in the output. This isn't prompt engineering for its own sake; it's defining the contract between me and the agent so I can reliably parse the review downstream.

Step 3 ‚Äî run it

invoke the agentpython
agent.print_response(
    "Read paper.txt and write review.md. "
    "Then list files and show the first 40 lines of review.md.",
    stream=True,
)

With stream=True, I can watch the agent reason in real time: it calls read_file, gets back the paper text, thinks for a moment, then calls write_file with the full review. Finally it calls list_files to confirm the output exists. That three-step trace is visible in the Colab output.

What the output looks like

review.md (excerpt)markdown
# Paper Review

## Summary
This paper introduces Denoising Diffusion Probabilistic Models (DDPMs), a class of
latent variable models inspired by non-equilibrium thermodynamics. The authors define
a fixed forward process that gradually adds Gaussian noise to data, and train a
reverse process ‚Äî parameterized by a neural network ‚Äî to iteratively remove it.
The main result is competitive image sample quality (FID 3.17 on CIFAR-10) combined
with strong log-likelihood estimates, something GANs typically cannot achieve.

## Contributions
- A simplified training objective (predicting noise rather than the mean) that
  outperforms the theoretically-motivated ELBO training.
- Progressive lossy compression interpretation of the generative process.
- State-of-the-art unconditional image synthesis on CIFAR-10 and 256√ó256 LSUN.

## Strengths
- Clean theoretical framework grounded in diffusion processes.
- The noise-prediction reparameterization is an elegant insight.
- Thorough ablation showing that the simplified loss drives the quality gain.

## Weaknesses
- Inference is slow: generating a single sample requires ~1000 denoising steps.
- No comparison to score-based generative models despite close conceptual overlap.
- Evaluation limited to CIFAR-10 and LSUN; no text-conditional or class-conditional
  experiments that would demonstrate broader applicability.

## Questions for Authors
- How does sample quality degrade if the number of inference steps is reduced to 50?
- Is the noise schedule learned or fixed? Could adaptive schedules improve quality?
- Have you tried the noise-prediction objective with continuous-time diffusion?

## Score: 8/10

That's a real output from the agent on the DDPM paper. It gets the key contributions right, identifies the slow inference weakness (which did become a major research direction post-publication), and asks a genuinely pointed question about step reduction ‚Äî exactly the kind of thing a reviewer should push on. Not bad for a model that was just handed a raw text file.

Lessons from building this

The biggest surprise was how much the instructions matter even at this simple level. My first version just said "review this paper" ‚Äî the output was long, unstructured, and not very useful. Once I specified the exact section headers and told the agent to "focus on empirical evaluation," quality jumped noticeably. The model isn't guessing at what I want; it's following a spec. Writing that spec well is the work.

The second lesson: stream=True is invaluable during development. Watching the agent call tools in sequence ‚Äî rather than just getting back a final response ‚Äî makes it immediately obvious when it's doing something wrong, like reading the wrong file or skipping the write step entirely.
